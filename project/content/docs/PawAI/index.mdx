# PawAI 語音互動系統 專題驗收報告總整理

## **一、 運作原理與系統流程**

**運作原理**：將使用者的自然語音轉換為機器狗大腦 (Brain) 可解析的文字與意圖事件，再將大腦生成的文字回覆，透過動態語音合成 (TTS) 與實體喇叭自然地播放出來。

- **運作流程**：
    1. **感測輸入**：透過外接 USB 麥克風進行實體收音。
    2. **語音轉意圖 (STT → Intent)**：截斷語句後，送至雲端轉換為文字，並分析出使用者隱含的動作意圖。
    3. **大腦決策 (Brain)**：語言模型接收意圖與文字，進行對話邏輯推論，決定回覆台詞與對應動作。
    4. **語音合成 (TTS)**：將台詞分派給雲端或本地模型生成音訊檔。
    5. **系統輸出**：透過 WebRTC 傳送至機器狗的外接 USB 喇叭發聲。

---

## **二、 系統架構與模型技術詳情**

語音系統是 PawAI 最核心的互動入口，承載了機器狗與使用者高達 70% 的互動比例。我們設計了一個具備高度容錯與備援機制的 11-stage 端到端 (End-to-End) 閉環架構，主要分為三大節點：

### **1. ASR 語音轉文字與意圖節點**

- **核心節點**：`stt_intent_node`
- **執行環境**：Jetson 邊緣運算平台（負責實體收音與本地模型運算）＋ RTX 8000 伺服器（負責雲端主力模型運算）
- **底層推論模型**：雲端 SenseVoice、本地 SenseVoice Local、本地 Faster-Whisper (int8 量化)
- **三層容錯架構 (Provider Chain)**：
    - **主力 (雲端)**：雲端 SenseVoice。將語音功能放在雲端的原因是運算速度極快（延遲僅 350-400ms），且辨識效果與準確度極高。
    - **次級 (邊緣)**：本地 SenseVoice Local。
    - **備援 (離線)**：本地 Faster-Whisper。確保網路完全斷線時，PawAI 仍具備基礎聽覺。
- **回音閘門 (Echo Gate)**：為防止「語音自激」（機器狗聽到自己講話而無限迴圈），系統會監聽 `/state/tts_playing` 狀態。發聲期間與結束後的 1.0 秒冷卻期內，強制拋棄麥克風收音。
- **抗幻覺機制**：針對 Whisper 在安靜環境下易產生「字幕by索蘭婭」等靜音幻覺，導入 OpenCC 繁簡轉換並設立黑名單過濾，節省後端運算資源。

### **2. 大腦決策中心**

- **核心節點**：`llm_bridge_node`
- **執行環境**：雲端 API（主副腦推論）＋ Jetson CPU（本地備案執行）
- **底層推論模型**：`gpt-5.4-mini` (主腦)、`gemini-3-flash` (副腦)、純規則模板 (RuleBrain)
- **主副腦架構**：主腦處理主要對話與決策；當主腦超時或異常時，無縫切換至副腦；若逢完全斷線，則由基於模板的 RuleBrain 進行規則防呆。
- **模型選型歷程**：開發期曾實測多款本地模型（如 Qwen 0.5B/2B/32B），但發現其在「講笑話、說故事」等擬真閒聊情境下語感生硬，最終決策採用雲端 LLM 以確保互動品質。

### **3. TTS 語音合成節點**

- **核心節點**：`tts_node`
- **執行環境**：雲端 API（雙軌路由）＋ Jetson CPU（本地 Piper 備援）
- **底層推論模型**：Gemini 3.1 Flash TTS (高品質)、Edge TTS (極速)、Piper 模型 (本地備援)
- **動態雙軌路由 (Dual-Route) 與本地備援**：
    - **高品質通道 (Quality Lane)**：針對 ≥12 字的長句，或帶有情緒標籤（如 `[excited]`）的句子，路由至雲端 Gemini 3.1 Flash TTS 進行高擬真情緒渲染。
    - **極速通道 (Fast Lane)**：針對「停」等安全關鍵字或 &lt;12 字短句，路由至雲端 Edge TTS，確保在 1.5 秒內做出緊急回應。
    - **本地備援**：在本地端配置了 Piper 模型，確保在網路不穩或斷線時，依然能夠透過本地算力完成語音合成發聲。
- **情緒繼承與平行切片 (Chunking)**：為解決長句生成過慢，系統將長句切片平行處理，並強制將開頭的情緒標籤複製到每一個 Chunk，確保語氣連貫。

## **三、 核心功能介紹**

- **自然語音喚醒與傾聽**：透過外接 USB 麥克風持續收音，並使用 Energy VAD 動態判斷語句段落。
- **意圖解析與肢體連動**：將使用者的語音轉為文字後，解析出隱含的動作意圖（如 `greet`, `sit`, `take_photo`），並同步觸發 Go2 機器狗的實體動作與語音回覆。
- **情感化對話陪伴**：支援情緒標籤解析，讓機器狗的回應不再是冰冷的機器聲，而是帶有喜怒哀樂的陪伴者。

## **五、 輸出資料規格 (LLM Contract)**

各節點間的溝通嚴格遵循 JSON 規格（定義於 `llm_contract.py`），確保大腦決策與硬體執行的精準對接。標準輸出格式如下：

```json
{
  "intent": "greet",              
  "reply_text": "[excited] 你好！我是 PawAI，你專專屬的居家互動機器狗！", 
  "selected_skill": "hello",      
  "reasoning": "使用者主動打招呼，應以開心語氣回應並做出打招呼動作。", 
  "confidence": 0.98              
}
```

## **六、 相關核心程式檔案清單**

- `stt_intent_node.py`: 負責收音、VAD 截斷、呼叫 ASR 轉譯及 Echo Gate 實作。
- `intent_classifier.py`: 負責初步意圖分類與 Whisper 幻覺黑名單過濾。
- `llm_bridge_node.py`: 管理大腦路由，處理 GPT、Gemini 與 RuleBrain 的切換邏輯。
- `tts_node.py`: 實作 TTS 雙軌路由、快取 (Cache) 機制與本地 Piper 備援、平行 Chunking。
- `vad_node.py`: 實作 Energy VAD 邏輯，控管靜音等待與緩衝時間。
- `llm_contract.py`: 定義系統資料結構與合約。

---

## **七、 系統現狀限制**

- **VAD 拖尾延遲 (效能瓶頸)**：目前使用基於音量的 Energy VAD，為了避免中斷使用者說話，設定了 800ms 靜音等待與 500ms 緩衝。這導致系統在使用者講完話後，需要 2~10 秒才能判定「收音結束」，是端到端互動最大的延遲來源。
- **Go2 喇叭數據通道例外錯誤 (Silent Fail)**：Go2 原廠 WebRTC DataChannel 不穩定，若發生錯誤時未強制送出 4002 (EXIT) 指令，會導致喇叭永久卡死。
- **外接硬體機構限制**：為解決原廠喇叭音質差 (16kHz) 與音量小的問題而改用外接 USB 喇叭。但當機器狗執行大動作時，USB 線路容易發生物理性鬆脫，導致系統報錯。

---

## **八、 未來優化方向**

- **升級 AI 神經網路 VAD**：計畫將 Energy VAD 汰換為基於神經網路的 VAD（如 Silero VAD），精準辨識「人聲」而非單純的「音量」，預計可將收音判斷延遲大幅縮短至 500ms 以內。
- **支援複合動作指令 (Composite Skills)**：修改 `llm_contract.py` 的規格限制，將單一動作指令升級為「動作陣列 (Action Array)」，希望可以同時觸發兩個動作。例如讓機器狗能夠一邊起立、一邊開心踏步或扭動身體，讓互動更流暢生動。
- **客製化 3D 列印硬體機構**：針對外接 USB 喇叭與麥克風，設計與列印專屬的固定座，從硬體層面解決走動時線路鬆脫的痛點。
