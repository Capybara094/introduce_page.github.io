# 各模型測試總整理

## 第一部分：ASR (語音辨識) 模型評測與選型路徑

ASR 的核心需求在於「中文辨識精準」、「低延遲」以及「對 Go2 機器狗運作時的風扇/馬達噪音有足夠的抗性」。

| **模型候選** | **來源 / 開源狀態** | **測試結果與特性摘要** | **決策狀態與原因** |
| --- | --- | --- | --- |
| **SenseVoice Cloud** | [GitHub: FunAudioLLM/SenseVoice](https://github.com/FunAudioLLM/SenseVoice) | 🏆 **準確度冠軍 / 情緒專家**<br />在 Go2 噪音測試 (A/B Test) 中，**正確率達 92%**，Intent 命中率 96%，延遲約 600ms。能精準處理中英夾雜且具備獨家情緒辨識。 | **✅ 最終主線**<br />準確率明顯優於 Whisper，解決了噪音環境下的辨識瓶頸。 |
| **SenseVoice Local (int8)** | 模型權重 (sherpa-onnx 格式) | 🚀 **極致輕量 / 穩定備援**<br />在 Jetson CPU 上執行，噪音測試正確率與 Intent 皆達 92%，延遲約 400ms，離線可用。 | **✅ 本地 Fallback**<br />使用 CPU int8 執行，完全不佔用 Jetson 的 GPU，符合多模組共存需求。 |
| **FunASR (Paraformer)** | [GitHub: modelscope/FunASR](https://github.com/modelscope/FunASR) | ⚡ **流式王者**<br />非自回歸設計，生成速度極快，能邊講邊出字，但 C++ 部署稍嫌繁瑣。 | **已淘汰**<br />在特定情境下有選字錯誤（例如將瓦斯爐辨識為「瓦斯魯」），穩定性不如 SenseVoice。 |
| **Faster-Whisper<br />(Large-v3/Small/Tiny)** | [GitHub: SYSTRAN/faster-whisper](https://github.com/SYSTRAN/faster-whisper) | ⚖️ **相容性極佳 / 早期主線**<br />Small/Tiny 版本曾在 3/21 benchmark 達到可用速度。 | **⚠️ 最終保底 (Fallback)**<br />在真機 Go2 噪音環境下暴露準確率瓶頸 (最佳僅 64%)，且容易在無聲處產生幻覺。 |
| **Sherpa-onnx** | [GitHub: k2-fsa/sherpa-onnx](https://github.com/k2-fsa/sherpa-onnx) | 🪶 **資源消耗極低 (14MB 模型)**<br />毫秒級反應，但容錯率低。 | **已淘汰**<br />在生活用語測試中出現「瓦斯 l l e」等嚴重幻覺亂碼。 |
| **Qwen3-ASR (0.6B)** | Qwen Research | 🧠 **語意修正強但太重**<br />能理解上下文修正錯字，但吃滿顯卡且延遲極高。 | **💀 不推薦**<br />推論太慢，出現亂碼與注音文。 |
| **Vosk** | [GitHub: alphacep/vosk-api](https://github.com/alphacep/vosk-api) | 🔋 **零延遲、極省電**<br />傳統架構，適合簡單指令喚醒。 | **已淘汰**<br />準確度上限較低，無法應付長篇中文對話。 |

## 第二部分：LLM (大腦決策) 模型評測與選型路徑

LLM 負責生成自然回覆與意圖判斷。核心需求是「速度快」、「輸出 JSON 格式極度穩定」，且「成本必須在可控範圍」。

| **模型候選** | **來源 / 部署方式** | **測試結果與特性摘要** | **決策狀態與原因** |
| --- | --- | --- | --- |
| **`openai/gpt-5.4-mini`** | 透過 OpenRouter 呼叫 | 👑 **綜合最佳**<br />P50 延遲僅 1.16s，P95 為 2.74s。成本低廉 ($0.018/12 calls)。 | **✅ 最終主線**<br />在 8-model A/B 測試中，唯一同時滿足速度、成本、格式穩定與角色自然度的模型。 |
| **`google/gemini-3-flash-preview`** | 雲端 API | 🥈 **便宜穩定**<br />P50 延遲 1.89s。曾在 5/4 評估中成為唯一穩定小於 2s 的主線。 | **✅ 雲端 Fallback**<br />便宜穩定，但在角色扮演自然度與 Context 掌握上稍弱於 gpt-mini。 |
| **RuleBrain** | 本地端規則引擎 | 🧱 **絕對安全網**<br />不依賴模型生成，由系統寫死的邏輯判斷。 | **✅ 最終保底 (Fallback)**<br />確保在所有 LLM 失效時，仍能維持基本安全與動作。 |
| **Qwen2.5-7B-Instruct** | 雲端 vLLM / 本地 | 🌟 **中文精通**<br />JSON 解析全通，P50 約 1.5s，曾為早期 (3/24) 雲端主線。但在 Jetson 8GB 上會 OOM (記憶體不足)。 | **已淘汰 (主線)**<br />後續被速度更快、更穩定的 gpt-mini 取代。 |
| **Qwen2.5-1.5B/0.5B** | Jetson Ollama | 📉 **輕量但有偏差**<br />1.5B 曾作本地 fallback，但意圖映射有偏差；0.5B 中文會漂移。 | **已淘汰**<br />品質不足以支撐展示。 |
| **Llama-3.2-3B (Taiwan)** | Hugging Face (微調版) | 🚀 **8GB Jetson 救星**<br />若硬性要求在 8GB Jetson 跑本地 LLM 的最佳解。 | **未進入最終主線**<br />因架構轉向依賴雲端主線 + 規則 Fallback，未實際導入。 |
| **Claude / DeepSeek 系列** | 透過 OpenRouter 呼叫 | 🔍 **各有缺陷**<br />Opus 成本過高；Haiku 會將 JSON 包在 markdown 中；DeepSeek reasoning 太慢 (P95 達 34s) 導致 demo 風險過高。 | **已淘汰**<br />未能通過 8-model A/B 測試的嚴格標準。 |

## 第三部分：TTS (語音合成) 模型評測與選型路徑

TTS 的核心需求是「音色自然有感情 (符合 PawAI 角色)」、「支援語氣標籤 (Audio Tag)」，並且對於「短句/安全警告」有極快的反應速度。

| **模型候選** | **來源 / 部署方式** | **測試結果與特性摘要** | **決策狀態與原因** |
| --- | --- | --- | --- |
| **Gemini 3.1 Flash TTS Preview<br />(Despina)** | 透過 OpenRouter Quality Lane | 🎭 **情感最豐富 (Quality Lane)**<br />延遲約 2.4s ~ 4.75s。對 `[excited]`, `[laughs]` 等 audio tag 原生渲染極佳。 | **✅ 最終主線 (長句/角色句)**<br />雖然犧牲了部分速度，但其中文音色與情緒標籤渲染明顯更適合 PawAI 的展示。 |
| **Edge-TTS** | Microsoft Azure<br />(開源客戶端套件) | ⚡ **極速與廣播級穩定**<br />P50 僅 1.13s，成功率 100%。不佔用本地 RAM。 | **✅ Fast Lane 主線**<br />保留給首音小於 2s 的短句、無 emotional tag 的句子，以及安全警告等需要極速反應的情境。 |
| **Piper (huayan)** | [GitHub: rhasspy/piper](https://github.com/rhasspy/piper) | 🔋 **離線穩定保底**<br />VITS / ONNX 架構，P50 約 2.03s，僅需 CPU，RAM 佔用極小。 | **✅ 本地 Fallback**<br />作為雲端服務失效時的最後一道防線，確保機器狗不斷線。 |
| **MeloTTS** | [GitHub: MyShell-ai/MeloTTS](https://github.com/myshell-ai/MeloTTS) | 👑 **本地端速度霸主**<br />完全離線且秒回，但中英夾雜需手動調教字典。 | **已棄用**<br />因有 ARM64 mecab 風險、PyTorch 依賴重，且需佔用 Jetson 500-800MB RAM，最終未採用。 |
| **CosyVoice** | [GitHub: FunAudioLLM/CosyVoice](https://github.com/FunAudioLLM/CosyVoice) | 🎨 **高可玩性但效能災難**<br />長句生成延遲高達 75s，硬體要求極高 (需高階 GPU)。 | **已淘汰**<br />無法滿足機器狗互動的即時性需求。 |
| **Qwen3-TTS (0.6B)** | Qwen Research | 🤕 **嚴重幻覺**<br />長序列生成中會崩潰，出現亂碼與注音文，速度極慢。 | **💀 不推薦**<br />模型未收斂，品質完全不可用。 |

## 總結：為何走向「雲端主力 + 本地 Fallback」架構？

回顧這些選型路徑，PawAI 團隊從最初嘗試「全本地運行 (Whisper + Qwen 1.5B/0.5B)」，逐漸演進到目前的「混合式架構」。核心原因在於硬體資源的殘酷現實：

> **「Jetson 8GB 必須與視覺 (物體辨識) 等模組共存，無法容納能提供高準確度與豐富情感的大型模型。」**

因此，最終確立了以下務實且具備高可用性的 **Dual-route routing (雙軌路由)** 策略：

1. **追求體驗極致 (主線走雲端)**：利用 `SenseVoice Cloud` 獲取高準確度與情緒，利用 `gpt-5.4-mini` 獲取聰明且穩定格式的大腦，再透過 `Gemini TTS` 渲染豐富的情感。
2. **確保生命線 (Fallback 走本地/規則)**：當遇到網路中斷或 API 失效時，立刻降級使用極輕量的 `SenseVoice Local (CPU)`、不依賴模型的 `RuleBrain`，以及完全離線的 `Piper TTS`，確保機器狗永遠不會「失聰」或「失語」。
